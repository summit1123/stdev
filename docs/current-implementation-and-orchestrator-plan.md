# Diary to Discovery 현재 구현 정리 및 오케스트레이터 기반 멀티 에이전트 승격안

## 문서 목적

이 문서는 현재 저장소에 실제로 구현되어 있는 기능과 구조를 기획자 관점에서 이해하기 쉽게 정리한 문서입니다.

특히 아래 두 가지를 명확히 구분합니다.

1. 지금 이미 동작하고 있는 구현
2. 앞으로 승격 가능한 아키텍처 방향

이 문서는 "이미 되어 있는 것"과 "이렇게 발전시키려는 것"을 섞지 않고 설명하는 데 목적이 있습니다.

---

## 한 줄 요약

현재 MVP는 아이의 손글씨 일기 이미지를 업로드하면, OCR 텍스트를 확인 및 수정한 뒤, 이를 바탕으로 질문 카드, 실험 카드, 과학 해설, 퀴즈/게임, 내레이션, 영상 시나리오, 결과 저장 및 다시 보기까지 이어지는 **스튜디오형 탐구 콘텐츠 생성 서비스**로 구현되어 있습니다.

다만 현재의 AI 구조는 **오케스트레이터 기반 멀티 에이전트**가 아니라, 하나의 planner 성격의 생성 호출이 결과 전체를 구조화해 내는 **단일 파이프라인 중심 구조**에 가깝습니다.

---

## 가장 먼저 바로잡아야 하는 표현

기획 문서나 대외 설명에서 아래처럼 쓰면 안 됩니다.

### 1. "OCR은 PaddleOCR을 사용한다"

현재 구현 기준으로 이 표현은 맞지 않습니다.

실제 구현은 OpenAI 비전 기반 OCR입니다.  
이미지 전체를 읽고, 본문이 충분히 읽히지 않으면 이미지를 상/중/하 band로 나눠 재인식하는 보강 로직까지 들어 있습니다.

즉 현재 표현은 아래가 맞습니다.

> 현재 OCR은 OpenAI 비전 기반 이미지 텍스트 추출을 사용하며, 긴 손글씨 일기의 본문 누락을 줄이기 위해 이미지 분할 재인식 보강 로직을 함께 적용하고 있다.

### 2. "지금도 멀티 에이전트 구조다"

이 표현도 현재 구현 기준으로는 맞지 않습니다.

지금은 OCR 이후에 하위 agent들이 병렬로 각자 질문/실험/퀴즈/게임/영상 시나리오를 따로 만드는 구조가 아니라, 하나의 콘텐츠 생성 단계가 전체 결과 스키마를 한 번에 만들어 내는 구조입니다.

즉 현재 표현은 아래가 맞습니다.

> 현재는 단일 planner 중심의 순차형 생성 파이프라인이며, 향후 OCR 구조화 결과를 입력으로 받는 오케스트레이터 기반 멀티 에이전트 구조로 승격 가능한 상태다.

---

## 제품 관점에서 본 현재 구현 상태

현재 제품은 단순한 데모 페이지가 아니라, 실제로 사용자가 업로드부터 결과 확인까지 따라갈 수 있는 **작업형 스튜디오 플로우**를 이미 갖추고 있습니다.

전체 흐름은 아래와 같습니다.

1. 일기 이미지 업로드
2. OCR 결과 확인 및 수정
3. 탐구 모드 선택
4. AI 결과 생성
5. 상황 이미지 생성
6. 내레이션 음성 생성
7. 영상용 스토리보드 및 mp4 생성
8. 결과 화면에서 카드/퀴즈/게임/실험/영상 확인
9. 관찰 로그 저장
10. 보관함에서 이전 세션 다시 열기

즉 현재 MVP는 "질문만 뽑아주는 기능"이 아니라, **입력 -> 해석 -> 콘텐츠 생성 -> 미디어 생성 -> 저장 -> 다시 보기**까지 이어지는 완성형 데모 구조에 가깝습니다.

---

## 현재 구현된 사용자 경험 상세

### 1. 업로드 화면

사용자는 첫 진입 후 업로드 스테이지에서 이미지 파일을 선택하거나 드래그 앤 드롭으로 업로드할 수 있습니다.

이 화면의 역할은 단순 파일 선택이 아니라 다음을 동시에 수행하는 것입니다.

- 이미지 입력 시작
- 원본 미리보기 노출
- OCR 단계로의 진입점 제공
- 이후 생성 흐름에 대한 기대치 전달

기획 포인트는 다음과 같습니다.

- 사용자는 "일기를 올리면 바로 읽힌다"는 반응을 즉시 받습니다.
- 원본 이미지가 곧바로 미리보기로 보여서 입력 신뢰감을 줍니다.
- 업로드 뒤 바로 OCR 확인 단계로 넘어가기 때문에 사용 흐름이 끊기지 않습니다.

### 2. OCR 확인 화면

OCR 확인 화면은 현재 MVP에서 매우 중요한 단계입니다.

이 단계에서 사용자는 다음을 수행합니다.

- 자동 추출된 텍스트 확인
- 오독 수정
- 원본 일기 이미지와 OCR 텍스트 비교
- warning 확인
- 탐구 모드 선택

여기서 중요한 UX 철학은 "AI가 읽은 결과를 그대로 강요하지 않는다"는 점입니다.

즉 시스템은 OCR을 자동화하지만, 최종 원문은 사용자가 확정합니다.  
이는 손글씨 일기 서비스에서 매우 중요합니다. 왜냐하면 손글씨 오독이 후속 질문, 실험, 미디어 생성 결과 전반을 흔들 수 있기 때문입니다.

### 3. 탐구 모드 선택

현재 구조에서는 OCR 단계에서 탐구 톤을 함께 정할 수 있습니다.

모드는 3가지입니다.

- 관찰 모드
- 실험 모드
- 모형/상상 모드

이 선택은 단순 UI 장식이 아니라, 생성 시 `preferredModeId`로 서버에 전달됩니다.

즉 기획적으로는 다음처럼 설명할 수 있습니다.

> 같은 일기라도 아이가 어떤 탐구 리듬으로 들어갈지 정할 수 있게 했고, 관찰 중심/실험 중심/모형화 중심의 진입 방식이 달라질 수 있도록 설계했다.

### 4. 결과 생성 중 상태 노출

현재 프론트는 생성 상태를 꽤 세밀하게 노출합니다.

상태 단계는 대략 다음과 같습니다.

- created
- text_ready
- queued
- parsing
- planning
- rendering_image
- rendering_audio
- rendering_video
- completed
- failed

즉 사용자 입장에서는 "지금 멈췄는지", "질문을 만드는 중인지", "이미지나 영상을 만드는 중인지"가 보이게 되어 있습니다.

이것은 단순 로딩 스피너보다 훨씬 기획적으로 좋습니다.  
특히 멀티모달 생성 서비스에서는 사용자가 기다릴 이유를 이해해야 이탈이 줄어듭니다.

### 5. 결과 화면

결과 화면은 단일 텍스트 응답을 뿌리는 구조가 아니라, 여러 종류의 결과물을 한 화면에서 소비하도록 설계되어 있습니다.

현재 결과 화면에서 보이는 요소는 다음과 같습니다.

- 오늘 장면 요약
- 감정 태그
- 질문 씨앗 최대 3개
- 과학 해석 영상
- 원본 일기
- 상황 이미지
- 과학 해설 패널
- 퀴즈
- 게임
- 미니 실험 / 미션 루프
- 관찰 로그 입력 폼
- 이전 관찰 로그 목록
- 영상 시나리오 텍스트

즉 결과는 "긴 글"이 아니라, **카드/패널/미디어 중심으로 쪼개진 탐구 경험**으로 제공됩니다.

### 6. 보관함

현재는 세션을 저장하고, 보관함에서 다시 열 수 있습니다.

보관함에서 보여주는 정보는 아래와 같습니다.

- 대표 썸네일
- 생성 요약
- 생성 상태
- 관찰 로그 개수
- 날짜

즉 한 번 생성하고 끝나는 서비스가 아니라, **이전 결과를 다시 열고 이어보는 회귀 구조**를 이미 일부 갖추고 있습니다.

## 현재 실제 구현 기능 상세

## 1. 입력 계층

### 1-1. 이미지 업로드

현재 웹 UI는 이미지 업로드가 메인 진입입니다.

- 파일 선택
- 드래그 앤 드롭
- 원본 미리보기
- 업로드 직후 OCR 호출

이미지 업로드가 끝나면 서버는 해당 세션용 엔트리를 만들고, 원본 파일을 저장한 뒤 OCR을 수행합니다.

### 1-2. 음성 입력 기반

모델/서버 구조에는 음성 입력 타입(`voice`)이 정의돼 있습니다.

즉 백엔드 설계상으로는 아래가 준비되어 있습니다.

- 입력 타입 구분: image / voice / text
- 음성 파일일 경우 STT 호출
- 음성 결과를 동일한 구조의 정규 텍스트로 변환

다만 현재 프론트 UI는 실제로 이미지 업로드 중심이며, 음성 업로드 UX는 전면 구현 상태로 보기는 어렵습니다.

즉 기획 문서에는 아래처럼 쓰는 것이 안전합니다.

> 음성 일기 수용을 위한 백엔드 라우팅과 모델 계층은 준비돼 있으나, 현재 메인 사용자 경험은 이미지 업로드 중심이다.

---

## 2. OCR / 텍스트 정규화 계층

### 2-1. 현재 OCR 방식

현재 OCR은 OpenAI 비전 기반입니다.

동작 방식은 다음과 같습니다.

1. 업로드 이미지를 읽음
2. 해상도가 낮으면 확대 보정
3. 전체 이미지를 대상으로 OCR 수행
4. 결과가 너무 짧거나 불완전하면 이미지 조각을 나눠 재인식
5. 최종적으로 `DiaryParseOutput` 구조로 정리

여기에는 다음 필드가 포함됩니다.

- transcription
- normalizedText
- summaryHint
- emotionTags
- scienceBridgeHint
- parseWarnings

즉 OCR 단계의 목적은 단순 문자열 추출이 아니라, **후속 생성에 들어갈 텍스트를 정돈하고 warning까지 포함한 구조화된 입력 패킷을 만드는 것**입니다.

### 2-2. OCR 확인 UX

OCR 결과는 바로 편집할 수 있습니다.

이 단계의 기획 의미는 큽니다.

- 원본 손글씨의 오독 리스크를 사용자 수정으로 줄임
- AI 생성 전, 원문 확정 단계를 둠
- 이후 모든 결과의 품질을 앞단에서 안정화

이 구조 덕분에 "일기 채점"이 아니라 "원문 존중 -> 탐구 전환"이라는 제품 철학도 더 잘 살아납니다.

---

## 3. 콘텐츠 생성 계층

현재 콘텐츠 생성은 하나의 큰 planner 호출이 결과 전체를 구조화하는 형태입니다.

즉 아래 결과들이 별도 agent가 아니라, 하나의 통합 생성 응답으로 생산됩니다.

- summary
- emotions
- scienceLens
- questionSeeds
- experimentCard
- scientificInterpretation
- sceneVisual
- scienceGame
- scienceQuiz
- gameModes
- recommendedModeId
- videoDirector
- creativeExpansion
- guardianNote
- narration
- sceneCards
- media

### 3-1. 질문 씨앗

현재 구조는 질문을 최대 3개로 제한합니다.

기획적으로 이는 매우 좋습니다.

- 카드 수를 통제할 수 있음
- 아이가 과부하되지 않음
- 질문이 "많이"가 아니라 "명확히" 제공됨

### 3-2. 실험 카드

실험은 1개만 제공되도록 유도되고 있습니다.

실험 카드에는 아래가 포함됩니다.

- 실험 제목
- 가설
- 독립 변수
- 종속 변수
- 방법
- 기간
- 관찰 포인트

이 구조는 단순 미션 제안이 아니라, **초기 수준의 과학 탐구 포맷**을 갖춘 것입니다.

### 3-3. 과학 해설 패널

과학 해설은 아래 구성으로 정리됩니다.

- observation
- concept
- explanation
- measurementIdea
- safetyNote

이 덕분에 결과가 감상문으로 흐르지 않고, **현상 -> 개념 -> 설명 -> 측정 -> 안전** 흐름을 유지합니다.

### 3-4. 과학 게임

현재 결과에는 게임형 콘텐츠가 별도 구조로 포함됩니다.

- 게임 제목
- 전제
- 목표
- 플레이 단계
- 승리 조건
- AI 가이드

즉 "과학 질문"을 끝내는 대신, 아이가 직접 조작하듯 따라갈 수 있는 **규칙 기반 탐구 놀이**로 연결하는 형태입니다.

### 3-5. 과학 퀴즈

퀴즈는 1문항 구조입니다.

포함 항목은 아래와 같습니다.

- 제목
- 질문
- 보기 배열
- 정답 인덱스
- 해설

현재 UI에서는 실제로 사용자가 선택하고 정답 피드백을 받을 수 있습니다.

### 3-6. 모드별 게임 카드

결과에는 아래 3개 모드용 카드도 함께 생성됩니다.

- observe
- experiment
- imagine

각 카드에는 아래가 들어갑니다.

- id
- title
- hook
- mission
- reward

즉 이 구조는 나중에 같은 일기를 여러 톤으로 재탐색하는 기능 확장에 매우 유리합니다.

### 3-7. 4장 카드형 결과

sceneCards는 카드형 결과 핵심입니다.

현재 결과 예시에서는 다음처럼 들어갑니다.

- 현상 요약
- 질문 씨앗
- 미니 실험
- AI 해설

기획 관점에서는 이 부분이 바로 "긴 글이 아니라 카드형 콘텐츠 4장"이라는 제품 약속에 가장 직접적으로 연결됩니다.

## 4. 미디어 생성 계층

## 4-1. 상황 이미지

결과 구조 안에는 sceneVisual이 있고, 여기에는 이미지 생성용 prompt와 caption이 포함됩니다.

생성 후에는 실제 이미지 파일이 저장되고 결과 JSON에 URL이 연결됩니다.

즉 이미지 생성은 단순 참조 문구가 아니라, 실제 사용자 소비 결과물입니다.

### 4-2. TTS 내레이션

내레이션은 다음 구조를 가집니다.

- script
- audioUrl
- voice
- durationSec

즉 현재는 스크립트만 뽑는 것이 아니라, 음성 파일까지 생성하여 결과에 붙이는 구조입니다.

기획적으로는 아래처럼 설명할 수 있습니다.

> 결과 카드와 별도로, 아이가 과학 해설을 듣는 경험을 제공하기 위해 TTS 내레이션이 생성되며, 오디오 재생은 결과 미디어 경험의 일부로 연결된다.

### 4-3. 영상 시나리오와 스토리보드

videoDirector에는 아래가 포함됩니다.

- title
- concept
- visualStyle
- mixDirection
- scenarioText
- soraPrompt
- targetDurationSeconds
- shots

shots는 여러 장면으로 구성되며, 각 장면마다 아래가 존재합니다.

- sceneTitle
- subtitle
- visualPrompt
- durationSeconds

즉 현재는 "영상 결과"를 단순 mp4 파일 하나로 보는 게 아니라, **영상 생성 직전 단계까지 구조화된 시나리오 자산**을 보유하는 형태입니다.

### 4-4. 로컬 영상 합성

현재 구조에서는 shot 이미지를 기반으로 로컬 mp4를 합성합니다.

흐름은 다음과 같습니다.

1. shot별 이미지 준비
2. 스토리보드 png 저장
3. 이미지별 segment mp4 생성
4. ffmpeg로 stitch
5. narration mp3가 있으면 오디오 합성
6. 최종 story-video.mp4 저장

즉 "영상 생성 실패 시 서비스 전체가 멈추는 구조"가 아니라, 최소한 로컬 영상 렌더링 결과를 통해 결과 체험을 이어갈 수 있는 하이브리드입니다.

### 4-5. Sora 연동 상태

현재 Sora 관련 구현은 아래 특징을 갖습니다.

- Sora 프롬프트 txt 저장
- create 요청용 JSON 파일 저장
- live/dry-run 모드 분기
- 실제 Sora 결과 mp4 URL을 연결할 수 있는 필드 보유

다만 현재는 "Sora 기반 최종 결과가 항상 완성되어 붙는 구조"로 보기는 어렵고, **Sora 요청 생성과 로컬 영상 렌더링이 함께 존재하는 하이브리드 상태**라고 보는 것이 정확합니다.

## 5. 저장 및 다시 보기 계층

현재는 각 엔트리별로 폴더 단위 저장이 이뤄집니다.

저장 단위 예시는 아래와 같습니다.

- `entry.json`
- `result.json`
- `mission_logs.json`
- `original.*`
- `scene-visual.png`
- `narration.mp3`
- `storyboard-xx.png`
- `segment-xx.mp4`
- `story-video.mp4`
- `sora-prompt.txt`

즉 이 구조는 단순 "DB에 텍스트만 저장"하는 것이 아니라, **세션 중심의 멀티모달 산출물 아카이빙** 구조입니다.

기획적으로는 아래 의미가 있습니다.

- 이전 세션 다시 보기 가능
- 결과 재생산이 쉬움
- 후속 분석/리뷰/디버깅 가능
- 데모 운영에 강함

## 현재 결과 스키마가 갖는 의미

현재 결과 스키마는 이미 꽤 명시적입니다.

이 점은 매우 중요합니다.

왜냐하면 기획이 흔들리지 않게 해 주는 기준이 되기 때문입니다.

현재 결과는 대략 아래 블록으로 나눌 수 있습니다.

- 원문 관련 정보
- 질문 관련 정보
- 실험 관련 정보
- 해설 관련 정보
- 게임/퀴즈 관련 정보
- 이미지 관련 정보
- 영상 관련 정보
- 오디오 관련 정보
- 저장/회고 관련 정보

즉 지금 구조는 앞으로 기능이 커지더라도 아무 형태로나 붙이는 것이 아니라, **명확한 산출물 단위로 관리 가능한 상태**입니다.

이것은 향후 아래 확장에 유리합니다.

- 보호자 노트 고도화
- 에이전트 분업화
- 개인화 추천
- 재시도 정책 분리
- 품질 검수 단계 추가

## 현재 아키텍처를 기획 언어로 설명하면

현재 시스템은 아래 네 층으로 이해하면 가장 쉽습니다.

### 1. Input Layer

- 이미지 업로드
- OCR / STT
- 텍스트 정규화

### 2. Planner Layer

- 일기 전체를 읽고
- 질문 / 실험 / 해설 / 게임 / 퀴즈 / 내레이션 / 영상 시나리오를
- 하나의 구조화 결과로 생성

### 3. Media Render Layer

- 상황 이미지 생성
- TTS 생성
- 스토리보드 생성
- 로컬 영상 합성
- Sora 요청 생성

### 4. Persistence Layer

- 엔트리 저장
- 결과 저장
- 로그 저장
- 미디어 파일 저장
- 보관함 다시 보기

이 설명은 현재 구현 사실을 과장하지 않으면서도, 제품의 완성도를 꽤 잘 보여 줍니다.

## 현재 구조의 강점

### 1. 입력 후 결과까지 이어지는 흐름이 실제로 닫혀 있다

업로드에서 끝나지 않고, 수정, 생성, 결과 소비, 저장, 회고까지 이어집니다.

### 2. 결과가 카드형/패널형이라 소비성이 좋다

긴 설명문 하나보다, 아이와 보호자가 각각 원하는 부분을 골라 보기 쉽습니다.

### 3. 미디어가 텍스트 부록이 아니라 핵심 가치로 들어가 있다

상황 이미지, 음성, 영상이 모두 별도 산출물로 존재합니다.

### 4. 결과 스키마가 명확하다

이후 멀티 에이전트 승격, 품질 검수, 개인화, 저장 구조 확장에 유리합니다.

### 5. 데모 운영 안정성이 있다

테스트 코드, fallback 구조, 세션 저장 구조가 이미 존재합니다.

---

## 현재 구조의 한계와 주의점

### 1. 첫 화면이 완전한 작업도구 직행형은 아니다

현재 홈 화면도 존재합니다.

즉 "첫 화면부터 오직 작업도구"라고 아주 강하게 쓰면 실제 구현과 차이가 생길 수 있습니다.

### 2. 음성 일기 UX는 아직 이미지만큼 전면화되지 않았다

백엔드 기반은 있으나, 현재 메인 UX는 이미지 중심입니다.

### 3. OCR 엔진은 PaddleOCR이 아니다

앞서 강조했듯, 이것은 반드시 정확히 써야 합니다.

### 4. 현재 생성 구조는 멀티 에이전트가 아니다

하위 기능이 개념적으로 나뉘어 있을 뿐, 실행 구조는 단일 planner 중심입니다.

### 5. 워커 큐가 아니라 thread 기반 비동기 처리다

즉 운영 규모가 커질 경우 작업 큐, 재시도, timeout, 자원 제어 구조는 추가 승격이 필요합니다.

### 6. Sora 파이프라인은 완결형보다는 하이브리드 상태다

Sora request asset과 로컬 렌더링 결과가 같이 존재하므로, 기획 문구는 과장 없이 써야 합니다.

### 7. fallback은 있으나 기본값이 항상 켜져 있는 상태는 아니다

즉 "실패 시 자동 fallback 완전 보장"이라고 쓰기보다,
실패 대비 구조와 대체 결과 경로가 준비돼 있다고 표현하는 편이 정확합니다.

## 기획자에게 설명하기 좋은 현재 제품 정의

아래 문장은 지금 구현 상태를 비교적 정확하고 설득력 있게 설명합니다.

> Diary to Discovery는 아이의 손글씨 일기를 입력으로 받아, 원문 확인과 수정 단계를 거친 뒤, 오늘의 장면을 과학 질문, 미니 실험, 카드형 해설, 게임/퀴즈, 내레이션, 영상 시나리오로 전환하는 탐구형 콘텐츠 스튜디오다. 현재는 업로드부터 결과 저장과 다시 보기까지 하나의 세션 흐름으로 연결되어 있다.

또는 조금 더 기획 문서형으로 쓰면:

> 본 MVP는 일기 이미지를 읽고 끝나는 OCR 도구가 아니라, 감정이 실린 일상 기록을 과학적 탐구 루프로 전환하는 멀티모달 콘텐츠 생성 스튜디오를 목표로 하며, 현재 버전에서 업로드, OCR 수정, 질문/실험 생성, 상황 이미지, 음성 내레이션, 영상 시나리오, 결과 저장 및 보관함 다시 보기까지의 흐름이 동작한다.

## 오케스트레이터 기반 멀티 에이전트 구조로의 승격안

이제부터는 **현재 구현이 아니라, 다음 단계 아키텍처 제안안**입니다.

이 섹션은 "이렇게 되어 있다"가 아니라 "이렇게 승격하는 것이 가장 자연스럽다"는 방향으로 읽어야 합니다.

## 왜 멀티 에이전트 승격이 필요한가

현재 구조는 결과 전체를 한 번에 잘 만들어 내는 장점이 있습니다.

하지만 규모가 커지고 품질 제어가 중요해질수록 아래 문제가 생길 수 있습니다.

- 질문 품질과 실험 품질이 함께 흔들림
- 퀴즈/게임/영상 시나리오를 독립적으로 개선하기 어려움
- 실패한 일부 결과만 재생성하기 어려움
- 역할별 안전 검수와 품질 검수가 분리되지 않음
- 병렬화 여지가 적음

즉 앞으로는 "하나의 큰 생성"보다, **역할별 생성과 병합** 구조가 더 적합합니다.

---

## 제안 구조 개요

제안하는 상위 구조는 다음과 같습니다.

1. OCR / 입력 정규화
2. Scenario Orchestrator
3. 하위 전문 Agent 병렬 호출
4. Merge / Validate
5. Media Workers
6. Save / Replay

---

## 1. OCR / 입력 정규화 단계

이 단계의 산출물은 단순 텍스트가 아니라, 하나의 공통 입력 패킷이어야 합니다.

예를 들어 다음과 같은 구조를 가질 수 있습니다.

- originalText
- normalizedText
- parseWarnings
- diaryMetadata
- candidateScienceSignals
- preferredMode
- riskFlags

이 공통 패킷을 여기서는 가칭 **Diary Evidence Packet**이라고 부를 수 있습니다.

즉 앞으로의 모든 하위 agent는 같은 입력을 공유하고, 서로 다른 책임으로 결과를 생성하게 됩니다.

---

## 2. Scenario Orchestrator

오케스트레이터는 멀티 에이전트 구조의 중심입니다.

이 계층의 역할은 아래와 같습니다.

- OCR 결과 수신
- 중심 과학 축 결정
- 탐구 모드 우선순위 결정
- 어떤 agent를 호출할지 결정
- 각 agent에 필요한 입력만 배분
- 병렬 실행 관리
- 부분 실패 감지
- 최종 결과 병합

즉 오케스트레이터는 스스로 모든 콘텐츠를 만들지 않고, **누가 무엇을 만들지 설계하고 조합하는 총감독** 역할을 맡습니다.

---

## 3. 하위 전문 Agent 구성안

현재 결과 스키마를 기준으로 하면, 아래처럼 agent를 분리하는 것이 가장 자연스럽습니다.

### 3-1. Question Seed Agent

역할:

- 질문 씨앗 최대 3개 생성
- 관찰 가능 / 비교 가능 질문만 통과
- 추상적 감상 질문 제거

출력:

- `questionSeeds`

### 3-2. Experiment Agent

역할:

- 실험 1개 설계
- 독립 변수 / 종속 변수 명시
- 기록 방법을 셀 수 있는 단위로 정리
- 안전성 검토

출력:

- `experimentCard`

### 3-3. Scientific Interpretation Agent

역할:

- 현상, 개념, 메커니즘, 측정 포인트를 구조화
- 감상문 톤 제거
- 관찰 기반 설명 유지

출력:

- `scientificInterpretation`
- `scienceLens`

### 3-4. Science Game Agent

역할:

- 즉시 해볼 수 있는 규칙형 게임 설계
- 관찰/비교/측정 중심 플레이로 정리

출력:

- `scienceGame`
- `gameModes`

### 3-5. Quiz Agent

역할:

- 1문항 관찰 퀴즈 생성
- 보기, 정답, 해설 완성

출력:

- `scienceQuiz`

### 3-6. Narration Agent

역할:

- TTS에 적합한 짧은 내레이션 작성
- 아이가 이해하기 쉬운 밀도 유지
- 영상 길이와 싱크 맞춤

출력:

- `narration.script`

### 3-7. Visual Scenario Agent

역할:

- 상황 이미지 프롬프트 생성
- 영상 샷 구조 생성
- 장면별 시각 단서 강화

출력:

- `sceneVisual`
- `videoDirector`

### 3-8. Safety / Policy Agent

역할:

- 진단/판정/낙인 표현 제거
- under-18 safe 검수
- 위험한 실험 제거
- 문장 난이도 점검

출력:

- risk report
- block / revise decision

## 4. 병렬 실행 개념

오케스트레이터는 OCR 구조화 결과를 주입받은 뒤, 아래처럼 병렬 실행할 수 있습니다.

### 병렬 묶음 A: 콘텐츠 설계

- Question Seed Agent
- Experiment Agent
- Scientific Interpretation Agent
- Science Game Agent
- Quiz Agent

### 병렬 묶음 B: 미디어 설계

- Narration Agent
- Visual Scenario Agent

### 검수 묶음 C: 정책/안전

- Safety / Policy Agent

그리고 마지막에 오케스트레이터가 전부 취합합니다.

즉 핵심 개념은 아래와 같습니다.

> OCR 결과를 단일 planner에 던져 전체를 한 번에 생성하는 대신, 오케스트레이터가 구조화된 일기 정보를 하위 기능 agent들에 주입하고, 각 agent는 자기 책임 영역만 생성한 뒤, 그 결과를 다시 오케스트레이터가 병합해 하나의 사용자 결과물로 완성한다.

---

## 5. Merge / Validate 단계

이 단계는 매우 중요합니다.

하위 agent 결과가 아무리 좋아도 서로 어긋나면 제품 품질은 오히려 나빠집니다.

오케스트레이터는 병합 시 아래 규칙을 강제해야 합니다.

- 질문은 3개 이하
- 실험은 1개만 허용
- 질문과 실험의 변수 논리가 이어져야 함
- 퀴즈와 해설의 개념이 충돌하지 않아야 함
- 내레이션과 영상 시나리오 길이가 맞아야 함
- 위험하거나 낙인적인 표현 제거
- 초등 고학년 수준 문장 유지

즉 멀티 에이전트 구조의 핵심은 agent 수가 많다는 것이 아니라, **오케스트레이터가 결과 정합성을 끝까지 책임진다**는 점입니다.

---

## 6. Media Worker 층 분리

콘텐츠 agent 층과 미디어 생성 층은 분리하는 것이 좋습니다.

이유는 다음과 같습니다.

- 텍스트 결과와 미디어 결과의 실패 유형이 다름
- 이미지/TTS/영상은 재시도 정책이 다름
- 일부 실패해도 카드 결과는 유지해야 함

따라서 미래 구조는 아래처럼 가는 것이 좋습니다.

### 콘텐츠 확정 후 실행되는 미디어 worker

- Scene Image Worker
- TTS Worker
- Storyboard Worker
- Sora Worker
- Local Render Worker

이 구조라면 "질문/실험/해설은 이미 완성"된 상태에서, 미디어만 부분 재시도할 수 있습니다.

---

## 7. 이 승격안이 현재 코드와 잘 맞는 이유

현재 코드베이스는 멀티 에이전트가 아니지만, 멀티 에이전트로 가기 좋은 전조가 이미 있습니다.

### 이유 1. 결과 스키마가 이미 분리돼 있다

질문, 실험, 해설, 퀴즈, 게임, 이미지, 영상, 내레이션이 하나의 텍스트 blob이 아니라 구조적으로 분리돼 있습니다.

### 이유 2. 미디어 생성이 이미 후단으로 분리돼 있다

텍스트 결과 생성과 이미지/TTS/영상 생성이 현재도 단계적으로 나뉘어 있습니다.

### 이유 3. 상태 단계가 이미 존재한다

queued, planning, rendering_image, rendering_audio, rendering_video 같은 상태가 있어, 이후 agent/worker 단계로 확장하기 좋습니다.

### 이유 4. 저장 단위가 세션 중심이다

각 세션 폴더 안에 결과와 미디어를 모으는 구조라, agent별 intermediate asset 저장도 확장하기 쉽습니다.

즉 현재 구조는 "멀티 에이전트를 새로 뜯어고쳐야 하는 상태"라기보다, **단일 planner 중심 구조를 orchestrated agent 구조로 점진 분해할 수 있는 상태**입니다.

---

## 기획 문서에 넣기 좋은 승격 설명 문안

아래 문안은 바로 가져다 쓸 수 있습니다.

> 현재 버전은 OCR 이후 단일 planner가 질문, 실험, 해설, 퀴즈, 게임, 내레이션, 영상 시나리오를 통합 생성하는 구조다. 다음 단계에서는 OCR 구조화 결과를 공통 입력 패킷으로 만든 뒤, Scenario Orchestrator가 이를 주입받아 질문 생성, 실험 설계, 과학 해설, 게임, 퀴즈, 내레이션, 비주얼 시나리오를 담당하는 하위 전문 agent들을 병렬 호출하고, 각 agent의 산출물을 안전 규칙과 형식 규칙에 맞게 병합하여 최종 결과를 조립하는 오케스트레이터 기반 멀티 에이전트 구조로 승격한다.

조금 더 짧은 버전도 가능합니다.

> 향후 아키텍처는 OCR 결과를 단일 생성 모델에 일괄 전달하는 방식에서 벗어나, 상위 오케스트레이터가 하위 전문 agent들을 병렬 호출해 질문, 실험, 해설, 게임, 퀴즈, 미디어 시나리오를 분산 생성하고, 이를 정합성 검수 후 하나의 결과 스키마로 병합하는 방식으로 진화한다.
