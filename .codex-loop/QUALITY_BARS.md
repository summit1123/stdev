# Quality Bars

The active profile is selected by `.codex-loop/config.json` under `loop.quality_profile`.
If it is blank, map `proposal`, `planning`, `submission`, and `contest` modes to `proposal`; map `prd` to `prd`; map `product-ui`, `ui`, `ux`, and `design` to `product-ui`; map everything else to `development`.
Completion requires satisfying every MUST item in the active profile.

## Development

Use this when the goal is shipping code, fixing behavior, or delivering a runnable product slice.

### MUST
1. Outcome reality
- A meaningful workflow, bugfix, or feature slice works end to end.
- Do not declare completion on scaffolding alone when a runnable slice is possible.

2. Verification evidence
- Relevant tests, checks, or screenshots actually ran and passed.
- If the change is UI-facing, include rendered evidence, not only source code edits.

3. Product quality
- Handle the important runtime states for the changed behavior: success, failure, and any obvious loading or empty state.
- Do not leave critical TODOs, placeholder copy, fake data, or manual steps unless the user explicitly accepted them.

4. Design quality
- If a UI exists, it must feel intentional rather than autogenerated.
- Fail generic card spam, weak hierarchy, sparse layouts, or obviously unfinished polish.

5. Integration truthfulness
- Update contracts, docs, config, migrations, or supporting files when behavior changed.
- Claims in docs and summaries must match the actual repo state.

### FAIL conditions
- Checks were skipped or are weak for the risk level.
- The main user path is still incomplete.
- The UI still looks generic or unfinished.
- Completion is claimed while important gaps are hidden in notes or TODOs.

## Proposal

Use this when the goal is a PRD package, pitch, submission document, strategy memo, or deck-like attachment.

### MUST
1. Source truthfulness
- The real narrative lives in Markdown source, not only in the final PDF.
- Source review must pass before packaging is treated as complete.

2. Reviewer-facing substance
- State the target reviewer, operator, or buyer.
- Explain the problem, urgency, solution, feasibility, business path, and expected effect in language a reviewer can reuse quickly.
- Remove assistant-style narration, filler claims, and unsupported hype.

3. Evidence structure
- Include the tables, comparisons, workflow descriptions, captures, or references needed for quick verification.
- If urgency depends on policy, market, regulatory, or news context, include source-backed evidence instead of unsupported copy.

4. Page density and composition
- Pages must feel authored, not template-generic.
- No thin pages, sparse spreads, empty boxes, decorative circles, or ornamental accents doing the narrative work.
- Use tables when they improve alignment; do not scatter related statements into floating fragments.

5. Submission portability
- Provide reusable one-line summary, keywords, solution summary, feasibility, business path, and effect statements.
- The output should be easy to move into a form, PDF, or presentation without rewriting the logic.

### FAIL conditions
- The Markdown source is thin and the layout is compensating for it.
- Necessary evidence, comparison, or workflow explanation is missing.
- The document still sounds like an assistant speaking to the user.
- PDF looks polished but source truth is weak or blocked.

## PRD

Use this when the goal is locking requirements, users, constraints, and the executable task graph before build work.

### MUST
1. Problem and user clarity
- Name the user, operator, or buyer and the real workflow being changed.
- Separate current pain from desired outcome.

2. Scope discipline
- Distinguish must-have scope, deferred scope, and explicit non-goals.
- Record assumptions instead of hiding them.

3. Executable requirements
- Requirements are concrete enough to build and verify.
- Acceptance criteria, dependencies, and constraints are written down.

4. Task readiness
- PRD, summary, and task graph agree with one another.
- There is one clearly runnable next task.

### FAIL conditions
- The PRD reads like brainstorming, not a build contract.
- Requirements are vague, contradictory, or untestable.
- Tasks do not match the current brief.

## Product UI

Use this when the goal is screen design, interaction flow, or polished front-end product delivery.

### MUST
1. Flow first
- The primary user path is clear before decorative detail.
- Screen structure and state transitions are explicit.

2. Visual system
- The design contract is specific about spacing, hierarchy, assets, banned patterns, and the selected reference pack.
- The UI does not look like generic AI output.
- Fail common generic patterns early: nested cards, ornamental gradients, weak contrast, and decorative chrome without workflow value.

3. Asset grounding
- Use approved assets, screenshots, references, or generated inputs intentionally.
- Keep the selected reference pack aligned with those assets instead of mixing unrelated styles.
- Record what each asset is for and whether it is approved for proposal, app, or both.

4. Verification
- Include screenshots or runtime evidence for the actual UI states touched.
- Text fits, layouts hold, and the main screen is not sparse or placeholder-like.

### FAIL conditions
- Card spam, ornamental circles, empty highlights, or split layouts are carrying the design.
- The screen still looks autogenerated.
- There is no asset or runtime evidence for the claimed design quality.
